## 设计

非常好的问题。经过前面的设计，我们已经覆盖了核心的解析落库、向量检索、Skill调用等环节，但要真正落地一个稳定、可扩展的生产级系统，还有以下几个**关键缺失和需要重点关注的地方**，每个都解释了为什么重要。

---

## 一、增量更新的准确性与一致性

### 缺失点
目前只讨论了“定时扫描近一年”或“基于 `last_commit_time` 增量”，但**没有处理 Git 历史被改写的情况**（如 `rebase`、`amend`、`force push`）。这些操作会导致已入库的 commit 消失或替换，造成图数据库和向量库中存在孤儿节点，且增量逻辑会漏掉新提交或重复处理。

### 为什么重要
- 团队使用 `rebase` 非常普遍，忽略会导致数据错误和查询结果不可信。
- 若不加处理，用户可能看到已废弃的提交，或者丢失最新的变更。

### 建议方案
- 每个仓库维护一个 **`branch_tip_history`** 表，记录默认分支（如 `main`）每次 `reflog` 更新的 commit SHA。
- 定期（如每小时）执行 `git reflog` 检测 tip 变化，若发生变化，则从上次记录的 tip 到当前 tip 之间的 commit 进行**全量重新解析**（因为 rebase 可能打乱顺序）。
- 对于孤儿 commit，通过图数据库查询并标记 `is_active=false`，在查询时过滤；同时定期物理删除。

---

## 二、大规模仓库的性能与资源控制

### 缺失点
设计中对性能考虑较少，尤其是：
- 大仓库（几万个 commit）全量解析时，逐个 commit 调用 JGit API 可能非常慢。
- `DiffFormatter` 默认会加载整个 diff 到内存，大文件 diff 可能导致 OOM。
- 同时解析多个 commit 时，没有控制并发度，可能耗尽文件句柄或内存。

### 为什么重要
- 若仓库有 5 万个 commit，按每秒处理 10 个计算，需要近 1.5 小时，且内存占用可能超过 2GB。
- 生产环境需要可预测的资源消耗和合理的处理时间。

### 建议方案
- **分页 + 批量提交**：每次只处理 1000 个 commit，然后记录 checkpoint。
- **控制并发**：使用固定线程池（如 CPU 核数），每个线程处理一个 commit 的 diff 解析和 AST 分析。
- **流式处理**：对于超大 diff（如 10MB+ 的 JSON 文件），不加载全部内容到内存，而是使用 JGit 的 `RawText` 按需读取。
- **限制 AST 解析范围**：只解析被修改的文件，且只解析被修改行附近的函数（而非整个文件）。

---

## 三、错误处理与可观测性

### 缺失点
目前代码示例中只有简单的 `try-catch` 和日志，但缺少：
- **解析失败的 commit 重试机制**（如网络超时、临时文件锁）。
- **部分失败时的状态保存**（例如某个 commit 的 diff 解析失败，不影响后续）。
- **监控指标**（处理速度、失败率、各阶段耗时）。

### 为什么重要
- 生产环境一定会遇到各种异常（仓库损坏、超大文件、权限问题等），没有容错机制会导致整个流程中断。
- 无法评估系统健康状况，也难以定位性能瓶颈。

### 建议方案
- 为每个 commit 的处理记录状态到一张 **`processing_state`** 表（如 MySQL 或 Redis），包含 `commit_hash`, `status` (pending/success/failed), `retry_count`, `error_msg`。
- 实现**死信队列**：失败超过 3 次的 commit 进入死信表，人工介入。
- 暴露 Prometheus 指标：`processed_commits_total`, `failed_commits_total`, `parse_duration_seconds`。

---

## 四、多语言 AST 解析的鲁棒性

### 缺失点
目前提到使用 Tree-sitter 或 JGit 自带的解析器，但实际实现中：
- 不同语言的语法差异巨大，且代码可能包含语法错误（在历史 commit 中常见）。
- 定位“diff hunk 属于哪个函数”需要精确的行号映射，但函数可能跨多个 hunk，或者 hunk 恰好落在函数外部（如文件头部）。

### 为什么重要
- 函数级别的语义检索是核心价值，如果函数归属错误，检索结果会严重降级。
- 历史代码可能含有语法错误，导致 AST 解析失败，需要降级策略。

### 建议方案
- 使用 **Tree-sitter**（支持增量解析、错误恢复）作为统一解析引擎，为每种语言提供语法。
- 对于无法解析的行，降级为**文件级**关系（即只知道 diff 改了文件，不知道具体函数）。
- 维护一个 **`parse_error_log`** 表，记录哪些文件/commit 解析失败，以便后续改进解析器。

---

## 五、图与向量数据的一致性

### 缺失点
我们设计了冗余字段（如 `File.lastCommitHash`、`DiffHunk.commitHash`）来加速查询，但**没有明确冗余字段的更新策略和一致性保证**。当增量更新时，如果某个 commit 被 rebase 删除，这些冗余字段不会自动更新，导致查询结果错误。

### 为什么重要
- 冗余字段一旦与真实关系不一致，查询会返回错误信息（例如显示某个文件最后一次修改是已删除的 commit）。
- 手动维护一致性代价高，容易遗漏。

### 建议方案
- **冗余字段仅用于展示和快速过滤，不用于关键决策**。核心查询仍应优先使用图关系。
- 增量更新时，如果检测到 commit 被移除，需要级联更新：
  - 找出该 commit 修改过的所有文件，重新计算每个文件的 `lastCommitHash`（取剩余 commit 中时间最新的）。
  - 删除对应的 `DiffHunk` 节点及其向量。
- 定期（如每周）运行**一致性校验脚本**，扫描冗余字段是否与图关系匹配，修复差异。

---

## 六、Skill 的 LLM 上下文优化

### 缺失点
Skill 会返回多个提交、多个文件和函数，但直接把这些原始数据扔给 LLM 会导致：
- **Token 爆炸**：一个功能可能关联 10+ 个提交，每个提交又包含多个文件的完整 diff，轻松超过 32k tokens。
- **信噪比低**：LLM 难以从大量重复或无关代码中提取关键信息。

### 为什么重要
- 用户体验差：LLM 响应慢，甚至因超长上下文而失败。
- 成本高：按 token 计费的模型会产生不必要的费用。

### 建议方案
- 在 Skill 中增加**智能摘要层**：
  - 对同一个文件的多次修改，只保留**最相关的 3 个 commit**（按相似度或时间）。
  - 对每个 diff hunk，只提取变更前后的 5 行关键代码，而不是整个函数。
  - 对 commit message 使用 LLM 生成 10 个字以内的**意图标签**（如“修复 NPE”、“增加缓存”）。
- 提供**分页/详情查询**接口：Skill 先返回摘要列表（commit hash + 标签），用户选择后再获取详细 diff。

---

## 七、安全性

### 缺失点
- 后端接口如果直接执行用户传入的 `file_path` 或 `commit_hash`，可能存在**路径遍历**或**注入攻击**（虽然 JGit 会校验，但 HTTP 接口未做白名单过滤）。
- 多租户场景下，用户可能越权访问其他仓库的数据。

### 为什么重要
- 生产环境的安全漏洞可能导致仓库源码泄露或系统被破坏。

### 建议方案
- 后端接口对 `file_path` 进行白名单校验（如正则 `^[a-zA-Z0-9_\-./]+$`），拒绝包含 `..` 或特殊字符的路径。
- 使用 **JWT 或 API Key** 进行认证，每个 token 绑定特定的 `repoId` 列表，查询时强制过滤 `repoId`（即使请求参数中未指定）。
- 对 `commit_hash` 使用 `ObjectId.isId(String)` 验证是否为有效的 SHA-1 格式。

---

## 八、总结：需要补充的设计文档

| 缺失项 | 优先级 | 建议文档/模块 |
|--------|--------|----------------|
| 增量一致性与 rebase 处理 | **高** | 增加 `ReflogMonitor` 模块，设计孤儿节点清理策略 |
| 性能与资源控制 | **高** | 设计分页处理、并发控制、内存限制参数 |
| 错误处理与可观测性 | **中** | 增加 `ProcessingState` 表、Prometheus 指标、死信队列 |
| AST 解析鲁棒性 | **高** | 引入 Tree-sitter，设计降级策略和错误日志 |
| 冗余字段一致性 | **中** | 设计一致性校验脚本和自动修复流程 |
| LLM 上下文优化 | **中** | 在 Skill 中实现智能摘要和分页加载 |
| 安全性 | **高** | 增加认证、授权、输入校验 |
